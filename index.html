Raytracer Manual

Introduction

A modular, plugin-oriented raytracing engine built in C++23. Rendering logic, geometry, scene management, output backends, and all renderable components are separated into independent layers connected through a runtime plugin system. New primitives, materials, lights, renderers, and interfaces can be added as shared libraries without touching the core.

Plugin Architecture

Every renderable component (primitive, material, light, renderer, interface) is loaded at runtime from a .so plugin via RayTracer::PluginManager. The plugin exposes an initializer map keyed by class name. RayTracer::SceneLoader calls RayTracer::PluginManager::getInitializer to retrieve constructors and populates the RayTracer::Scene.

Built-in plugins:

  • CorePlugin — RayTracer::Camera, RayTracer::SceneWriter, all standard primitives, materials, and lights
  • LivescenePlugin — RayTracer::Livescene, real-time SFML viewer

Rendering Pipeline

  1. Load scene: RayTracer::Parser reads the .cfg file into a RayTracer::Node tree.
  2. Build scene: RayTracer::SceneLoader uses RayTracer::PluginManager to instantiate a RayTracer::Camera, primitives, and lights into a RayTracer::Scene.
  3. Acceleration: if useBVH is enabled, RayTracer::Scene::prepareAccelerationStructure builds a RayTracer::BVH from bounded primitives; unbounded ones (planes) fall back to linear traversal.
  4. Execute: the chosen RayTracer::AInterface calls RayTracer::ARenderer::render.
  5. Render: RayTracer::Camera iterates every pixel with a stratified sample grid (√N × √N per pixel) and calls RayTracer::Camera::castRay for each sample.
  6. Trace: castRay recurses up to _maxDepth bounces — for each hit it evaluates the RayTracer::IMaterial BRDF (evaluate / sample / pdf), queries lights via RayTracer::ILight::computeLight, and accumulates radiance.
  7. Output: the filled framebuffer is written to a PPM file (RayTracer::SceneWriter) or displayed in an SFML window (RayTracer::Livescene).

Design Rationale

  • Dependency inversion — abstractions (RayTracer::ARenderer, RayTracer::APrimitive, RayTracer::IMaterial, RayTracer::ILight) never depend on concrete implementations; plugins depend on abstractions.
  • Stateless renderer — RayTracer::ARenderer::render returns a framebuffer; no I/O or side effects inside the renderer.
  • Output decoupled from rendering — RayTracer::AInterface handles all output; RayTracer::Camera stays a pure filter.
  • Scene owns acceleration — the RayTracer::BVH lives in RayTracer::Scene, not the renderer, keeping traversal state close to the primitive list. See Space Partitioning.
  • Runtime extensibility — third-party or experimental components load as .so files without recompiling the core. See Plugin System.
