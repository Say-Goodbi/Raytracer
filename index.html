Raytracer Manual

Introduction

This is a modular, plugin-oriented raytracing engine built in C++. It separates concerns across geometry, primitives, materials, and rendering, allowing extensibility through plugins while maintaining a clean separation of ray-casting logic from output handling.

Folder Organization

The repository is organized to separate project-level artifacts from core source code and implementations. High-level folders:

  • dev/ : Development helpers, scripts, and contribution guides.
  • docs/ : Documentation sources (Doxygen input, guides).
  • src/ : Primary C++ source tree (see detailed grouping below).

Within src/, code is organized by concern and role:

Core Framework (Interface Definitions)

These folders define the plugin architecture and core abstractions—they never depend on implementations:

  • Objects/ : Plugin system boundaries and abstract base classes
    • Abstracts/ : Core abstract interfaces (ARenderer, AInterface, APrimitive, IMaterial, ILight)
    • IPlugin.hpp : Plugin interface for runtime component factories
  • Exceptions/ : Custom exception types (domain errors, runtime problems)
  • Geometry/ : Fundamental math and geometric primitives
  • Utils/ : Small helpers (Color, etc.)
  • SceneSystem/ : Scene composition and plugin management

Plugin Implementation (CorePlugin)

The CorePlugin/ folder contains the built-in plugin providing essential raytracing functionality:

  • CorePlugin.hpp/cpp : Plugin entry point; provides component factories
  • Renderers/Camera/ : Camera implementation (ARenderer subclass)
  • Interfaces/SceneWriter/ : PPM file output (AInterface subclass)
  • Primitives/ : Renderable shapes (APrimitive subclasses)
  • Materials/ : Surface shaders (IMaterial subclasses)
  • Lights/ : Light sources (ILight subclasses, currently empty)

Design Rationale

Separation of Plugin System from Implementations:**

  • Objects/Abstracts/ defines the plugin contract (what plugins must implement)
  • CorePlugin/ is one concrete plugin implementation; other plugins would extend the same abstractions

  • This allows third-party or experimental plugins without modifying core code

    Layering:**

  • Geometric primitives (Geometry/) are independent of rendering logic
  • Abstract interfaces (Objects/) are independent of implementations
  • Implementations (CorePlugin/) depend on abstractions but abstractions don't depend on implementations
  • This follows dependency inversion and allows clean extension

Architecture Overview

The architecture is built on several core abstractions:

Rendering Pipeline: ARenderer → Camera

ARenderer is the abstract base for all renderers. It defines the interface:

  • render() — produces a 2D framebuffer

Camera is the concrete implementation of ARenderer. It:

  • Holds camera position, direction, and field-of-view (FOV)
  • Generates rays for each pixel via generateRay()
  • Casts rays into the scene and computes colors via castRay()
  • Handles shadow computation via computeShadow()

  • Returns the fully rendered framebuffer (without writing it)

    Why: Separating the renderer abstraction from output writing allows different rendering strategies (e.g., ray-tracing, path-tracing) and different output formats without duplication.

Output Pipeline: AInterface → SceneWriter

AInterface owns an ARenderer* and coordinates the overall application flow.

SceneWriter is a concrete AInterface that:

  • Takes a Camera (ARenderer) and output filename
  • Calls render() to get the framebuffer
  • Writes the framebuffer to a PPM file

  • Can be replaced by other interfaces (game loops, real-time viewers, etc.)

    Why: Output handling is separate from rendering—the renderer stays stateless and doesn't know about I/O.

Scene Graph

Scene holds:

  • A Camera (the active renderer)
  • A list of primitives (shapes to render)
  • A list of lights (light sources)

SceneLoader (planned) will parse scene descriptions and populate scenes programmatically.

Geometry

Core geometric types in the Geometry namespace:

  • Point3D — a 3D position
  • Vector3D — a 3D direction/magnitude; supports normalize(), length(), dot, cross
  • Ray — origin + direction, the fundamental unit cast through the scene
  • Rectangle3D — planar screen surface (defined by two edge vectors)
  • HitRecord — records intersection data: point, surface normal, ray distance, material

Primitives: Objects & Materials

APrimitive is the abstract shape base. Subclasses:

  • Plane — infinite flat surface
  • Sphere — sphere (planned)

Each primitive:

  • Stores an IMaterial* pointer
  • Implements hit() to test ray–primitive intersection (returns an optional HitRecord)
  • Via getMaterial(), provides shading instructions

IMaterial defines shading behavior:

  • computeColor() — computes pixel color given an intersection and lights

FlatColor is a simple material that returns a constant RGB value.

Why: Decoupling geometry from appearance (material) allows reuse and runtime material swapping.

Lights

ILight is the light abstraction:

  • getPosition() → Point3D (e.g., point light position)
  • computeContribution() — computes light's effect at a hit point

Lights are stored in the Scene and referenced during shading computations.

Plugin System & CorePlugin

Plugins are compiled as separate dynamic libraries (.so) and can provide:

  • APrimitive subclasses — custom shapes
  • IMaterial subclasses — custom shaders
  • ILight subclasses — custom light types
  • ARenderer subclasses — alternative renderers (e.g., path-tracer)
  • AInterface subclasses — application backends (game loop, network streaming, etc.)

PluginLoader handles runtime .so loading. PrimitiveFactory instantiates plugins on demand.

The CorePlugin is the built-in plugin providing essential components.

Rendering Pipeline

  1. User creates a Scene with Camera, primitives, and lights
  2. Create a Camera (ARenderer instance) with resolution and view parameters
  3. Wrap it in a SceneWriter (AInterface) with output path
  4. Call sceneWriter->execute(scene)
  5. SceneWriter calls camera->render(scene):
    • For each pixel (x, y):
      • Generate normalized coordinates (u, v)
      • Generate a ray via camera->generateRay(u, v)
      • Cast the ray: castRay(ray, scene) returns a Color
    • For each ray, find the closest primitive intersection
    • If hit, compute shadow from lights, fetch material, shade via material->computeColor()
    • Return 2D framebuffer
  6. SceneWriter writes framebuffer to PPM file

Design Rationale

  • Abstraction hierarchy — ARenderer, AInterface, APrimitive, IMaterial, ILight provide clear extension points
  • Immutable rendering — render() returns a framebuffer; no side effects
  • Separation of concerns — rendering logic (Camera) is independent of I/O (SceneWriter)
  • Plugin architecture — allows closed-source or experimental code without modifying core
  • No output in Camera — the Camera class is a pure filter; output is delegated

Future Enhancements

  • Path tracing (ARenderer subclass)
  • Spheres and complex primitives
  • Texture mapping (IMaterial extensions)
  • Spatial acceleration structures (BVH, KD-trees)
  • Parallel rendering (multithreaded Camera)
  • Real-time viewer (AInterface subclass)
  • Network rendering (distributed AInterface)
